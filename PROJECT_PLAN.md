# 🏃 孩子运动健康推荐系统 - 项目规划

**文档版本**: v4.0 (竞品迭代版)
**最后更新**: 2026-03-26
**迭代轮次**: 15 轮 (7 轮通用 +5 轮公益 +3 轮竞品)
**项目性质**: 公益项目 (永久免费)

---

## 📋 项目概述

**项目名称**: ChildFit - 看天安排孩子运动的公益 App

**项目性质**: 🎗️ 公益项目 (所有功能永久免费，无付费墙、无广告)

**核心愿景**: 根据天气、学生作息表、年龄段，动态安排个性化的活动时间和饮食推荐，培养孩子健康生活习惯。

**公益使命**: 让每个孩子——无论城市农村、富裕贫困——都能获得科学的健康指导，养成受益终身的运动习惯。

**Slogan**: "雾霾天不知道让孩子干嘛？用它！"

**副 Slogan**: "所有功能永久免费，农村城市同等可用"

**核心差异化** (竞品都没有的):
| 差异化 | 说明 | 竞品状态 |
|--------|------|----------|
| 🌤️ **天气动态适配** | 雾霾/雨天自动室内，晴天优先户外 | 所有竞品都没有 |
| 👁️ **运动 + 护眼组合** | 每日户外 2 小时 + 眼保健操 + 视力追踪 | 单一维度有，整合的没有 |
| 📅 **作息智能联动** | 上学日/周末/考试周自动适配 | 所有竞品都没有 |
| 🎗️ **公益 + 普惠** | 永久免费 + 农村适配 + 特殊儿童关怀 | 商业竞品做不到 |

**竞品对比一句话**: Keep 做课程深度，咕咚做硬件联动，天天跳绳做学校场景，**我们做天气 + 作息 + 护眼三维联动**。

## 🎯 核心功能模块

### 1️⃣ 用户档案系统

```
├── 孩子基本信息
│   ├── 年龄/出生日期 (自动计算年龄段)
│   ├── 性别
│   ├── 身高体重 (BMI 计算)
│   ├── 健康状况 (过敏史、慢性病等)
│   └── 运动能力评估
│
├── 年龄段划分
│   ├── 幼儿组 (3-6 岁)
│   ├── 小学低年级 (7-9 岁)
│   ├── 小学高年级 (10-12 岁)
│   └── 青少年 (13-18 岁)
│
└── 家庭信息
    ├── 所在城市 (用于天气获取)
    ├── 居住环境 (小区/公园/运动设施)
    └── 家长联系方式
```

### 2️⃣ 智能天气引擎

```
├── 天气数据接入
│   ├── 实时天气 (温度、湿度、AQI)
│   ├── 天气预报 (24h/7 天)
│   ├── 特殊天气预警
│   └── 紫外线指数
│
├── 天气规则引擎
│   ├── 雾霾天 (AQI>150) → 室内活动
│   ├── 晴天 (AQI<50, 紫外线适中) → 户外活动
│   ├── 雨天 → 室内活动/雨具户外
│   ├── 高温 (>35°C) → 清晨/傍晚户外 + 防晒
│   ├── 低温 (<5°C) → 保暖户外/室内
│   └── 紫外线强 → 防晒措施 + 限时户外
│
└── 季节适配
    ├── 春季：过敏原提醒
    ├── 夏季：防晒 + 补水 + 防暑
    ├── 秋季：温差适应
    └── 冬季：保暖 + 室内备选
```

### 3️⃣ 作息时间表管理

```
├── 日程类型
│   ├── 上学日 (周一至周五)
│   ├── 周末 (周六、周日)
│   ├── 寒暑假
│   └── 节假日
│
├── 时间块划分
│   ├── 晨间 (6:00-8:00)
│   ├── 放学后 (16:00-18:00)
│   ├── 晚间 (19:00-21:00)
│   └── 周末全天
│
└── 学校作息同步
    ├── 上学/放学时间
    ├── 课间休息
    ├── 体育课安排
    └── 作业时间预估
```

### 4️⃣ 运动活动推荐引擎

```
├── 活动库 (按年龄段分类)
│   │
│   ├── 幼儿组 (3-6 岁)
│   │   ├── 户外：滑梯、秋千、追泡泡、平衡车
│   │   ├── 室内：积木、舞蹈、亲子游戏
│   │   └── 时长：每次 15-30 分钟
│   │
│   ├── 小学低年级 (7-9 岁)
│   │   ├── 户外：跳绳、跑步、自行车、足球
│   │   ├── 室内：乒乓球、游泳、体操
│   │   └── 时长：每次 30-45 分钟
│   │
│   ├── 小学高年级 (10-12 岁)
│   │   ├── 户外：篮球、羽毛球、轮滑、游泳
│   │   ├── 室内：健身操、瑜伽、力量训练
│   │   └── 时长：每次 45-60 分钟
│   │
│   └── 青少年 (13-18 岁)
│       ├── 户外：跑步、球类运动、健身
│       ├── 室内：健身房、HIIT、舞蹈
│       └── 时长：每次 60-90 分钟
│
├── 动态推荐逻辑
│   │
│   ├── IF 雾霾天 THEN 室内活动列表
│   ├── IF 晴天 AND 紫外线适中 THEN 户外活动优先
│   ├── IF 高温 THEN 清晨/傍晚时段 + 防晒提醒
│   ├── IF 雨天 THEN 室内活动 OR 雨中散步 (可选)
│   └── IF 连续室内 3 天 THEN 提醒安排户外补偿
│
└── 活动难度递进
    ├── 新手级 → 进阶级 → 挑战级
    └── 根据完成情况自动调整
```

### 5️⃣ 眼保健与视力保护

```
├── 眼保健操
│   ├── 标准版 (学校版)
│   ├── 简化版 (5 分钟快速)
│   └── 视频跟练指导
│
├── 用眼提醒
│   ├── 20-20-20 规则 (每 20 分钟看 20 英尺外 20 秒)
│   ├── 屏幕时间统计
│   └── 定时休息提醒
│
├── 户外活动护眼
│   ├── 每日户外 2 小时目标
│   ├── 自然光暴露追踪
│   └── 近视防控建议
│
└── 视力记录
    ├── 定期视力检测记录
    ├── 趋势分析
    └── 异常预警
```

### 6️⃣ 饮食营养推荐

```
├── 营养需求计算
│   ├── 基础代谢率 (BMR)
│   ├── 活动消耗估算
│   ├── 年龄段营养标准
│   └── 生长发育需求
│
├── 食谱推荐
│   ├── 早餐搭配 (上学日/周末)
│   ├── 午餐建议 (学校/家庭)
│   ├── 晚餐搭配
│   └── 加餐/零食建议
│
├── 季节饮食调整
│   ├── 夏季：清淡 + 补水 + 防暑食物
│   ├── 冬季：温热 + 高能量 + 免疫支持
│   ├── 春季：助长 + 维生素补充
│   └── 秋季：润燥 + 营养储备
│
├── 运动前后饮食
│   ├── 运动前：能量补充 (提前 30-60 分钟)
│   └── 运动后：蛋白质 + 碳水恢复
│
└── 饮食禁忌
    ├── 过敏原标记
    ├── 不推荐食物
    └── 摄入量限制
```

### 7️⃣ 打卡与激励系统

## 🏗️ 技术架构

### 整体架构

```
用户层 (微信小程序 / H5)
        ↓
API 网关 (限流、鉴权、日志)
        ↓
业务服务层
├── 用户服务 (档案、认证)
├── 天气服务 (多源融合、缓存)
├── 推荐服务 (规则引擎 + ML)
├── 打卡服务 (AI 识别、记录)
├── 通知服务 (模板消息、推送)
└── 数据服务 (统计、报表)
        ↓
数据层
├── PostgreSQL (用户/活动/记录)
├── Redis (缓存/会话/排行榜)
├── MongoDB (行为日志)
└── OSS (图片/视频存储)
```

### 前端
```
├── 微信小程序 (主战场)
│   ├── 家长端 (完整功能)
│   └── 孩子端 (卡通简化版)
├── H5 页面
│   ├── 快速分享页
│   └── 活动成果展示
└── 可选：App (后期)
    ├── React Native / Flutter
    └── 功能与小程序同步
```

### 后端
```
├── 语言：Node.js (TypeScript) / Python
├── 框架：NestJS / FastAPI
├── 数据库：
│   ├── PostgreSQL (主库 - 用户/活动/记录)
│   ├── Redis (缓存 - 天气/推荐结果/会话)
│   └── MongoDB (日志 - 用户行为/打卡记录)
├── 天气 API:
│   ├── 和风天气 (主)
│   ├── 心知天气 (备)
│   └── 自动切换/融合
├── AI 服务:
│   ├── 打卡图片识别
│   └── 语音合成 (孩子端引导)
└── 部署：
    ├── Docker 容器化
    ├── Kubernetes (后期)
    └── 云服务器 (腾讯云/阿里云)
```

### 核心算法
```
├── 推荐引擎
│   ├── Phase 1: 规则引擎 (IF-THEN)
│   ├── Phase 2: 协同过滤 (相似用户)
│   └── Phase 3: 强化学习 (反馈优化)
├── 个性化排序
│   ├── 基于历史完成偏好
│   ├── 基于年龄段特征
│   └── 基于天气适配度
├── 异常检测
│   ├── 连续未运动预警
│   ├── 视力风险预警
│   └── 营养不均衡预警
└── A/B 测试框架
    ├── 推荐策略对比
    └── UI/文案优化
```

### 关键性能设计
```
├── 天气数据
│   ├── 定时预取 (每 30 分钟)
│   ├── 多城市缓存
│   └── 降级策略 (API 故障时用本地规则)
├── 推荐结果
│   ├── 同条件孩子复用
│   ├── 按时间段预计算
│   └── 缓存 TTL: 1 小时
├── 打卡高峰期
│   ├── 限流 (17:00-19:00)
│   ├── 异步处理
│   └── 队列缓冲
└── 监控告警
    ├── API 响应时间 <200ms
    ├── 错误率 <1%
    └── 活跃度异常告警
```

| 基金会资助 | 儿童健康/教育类基金会 (3-5 年周期) | 40% |
| 政府购买服务 | 卫健委/教育局采购服务 | 30% |
| 企业 CSR 合作 | 企业社会责任项目 (资金 + 技术) | 20% |
| 个人捐赠 | 月捐计划、一次性捐赠 | 10% |

**成本控制策略**:
- 云服务申请公益额度 (腾讯云/阿里云公益计划)
- 核心功能自研，非核心用开源方案
- 志愿者参与 (技术志愿者、专家顾问公益支持)
- 不做广告投放，靠口碑 + 合作获客

### 公益合作模式

| 合作方 | 合作内容 | 互利点 |

| 媒体 | 传播、故事报道 | 优质公益内容、社会影响力 |

## ⚠️ 风险与注意事项

### 健康安全 ⚕️

| 风险 | 应对措施 |
|------|----------|
| 运动受伤 | 活动难度分级 + 安全提示 + 免责声明 |
| 饮食不适 | 过敏原问卷 + 禁忌标记 + 不替代医疗建议 |
| 过度运动 | 每日时长上限 + 疲劳提醒 + 休息建议 |
| 特殊情况 | 生病/受伤手动暂停模式 + 复课过渡计划 |

**必须措施**:
- [ ] 法律条款请专业律师审核
- [ ] 健康问卷设计 (排除禁忌症)
- [ ] 建立专家顾问团 (儿科/运动医学/营养师)
- [ ] 明确免责声明 (不替代医疗建议)

### 隐私保护 🔒

| 风险 | 应对措施 |
|------|----------|
| 信息泄露 | 端到端加密 + 最小化收集 |
| 未授权访问 | 家长授权机制 + 登录验证 |
| 数据滥用 | 严格内部权限 + 审计日志 |

**必须措施**:
- [ ] 符合《儿童个人信息网络保护规定》
- [ ] 数据加密存储 (AES-256)
- [ ] 传输加密 (HTTPS/TLS)
- [ ] 家长授权确认 (每次收集前)
- [ ] 数据导出/删除权利

### 内容准确性 📚

| 领域 | 标准 |
|------|------|
| 天气数据 | 多源融合 + 自动切换 + 降级策略 |
| 营养推荐 | 参考《中国居民膳食指南 (儿童版)》 |
| 运动建议 | 参考 WHO 儿童运动指南 + 专家审核 |
| 视力保护 | 参考国家卫健委近视防控指南 |

**必须措施**:
- [ ] 活动库定期更新 (季度)
- [ ] 专家审核机制
- [ ] 用户反馈快速响应

### 运营风险 📉

| 风险 | 应对措施 |
|------|----------|
| 用户流失 | 持续内容更新 + 社交绑定 + 习惯养成 |

### 主要竞品对比

| 功能/特性 | Keep 青少年 | 咕咚 | 天天跳绳 | 护眼宝 | 学校平台 | **ChildFit** |
|-----------|-------------|------|----------|--------|----------|-------------|

| 运动 + 护眼 | ❌ | ❌ | ❌ | ✅ | ❌ | ✅ |
| 作息联动 | ❌ | ❌ | ❌ | ❌ | 部分 | ✅ |
| 年龄段划分 | 粗 | 粗 | 粗 | 无 | 按年级 | **细 (4 段)** |
| 公益免费 | ❌ | ❌ | ❌ | 部分 | ✅ | ✅ |
| 农村适配 | ❌ | ❌ | ❌ | ❌ | 部分 | ✅ |

### 竞品启示

**学习的**:
- Keep 的课程质量 → 活动视频专业标准
- 天天跳绳的学校合作 → 批量获客模式
- 护眼宝的专业内容 → 眼科医生顾问团
- Active Kids 的公益模式 → 影响力评估框架

**避免的**:
- 付费墙 (引起用户不满)
- 广告过多 (影响体验)
- 功能单一 (粘性低)
- 城市导向 (忽略农村)
- 体验差 (学校平台通病)

**差异化策略**:
- 不做 Keep 的课程深度 → 专注天气适配
- 不做咕咚的硬件联动 → 专注软件体验
- 不做天天跳绳的单一功能 → 专注整合方案
- **做竞品都没有的：天气 + 作息 + 护眼三维联动**
